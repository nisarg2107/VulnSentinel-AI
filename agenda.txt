PROJECT AGENDA: VulnSentinel AI
1. Project Overview & Objectives
• The Problem: Traditional vulnerability scanners generate massive amounts of noise and "alert fatigue" by flagging vulnerabilities without considering runtime context, such as whether a vulnerable component is actually exposed to the internet. Furthermore, scanning container images at scale (e.g., 100k - 1M images) strains network bandwidth and compute resources.
• The Solution: Build a local-only, enterprise-grade vulnerability management pipeline using a "Spec-Driven" workflow.
• Core Strategy:
    ◦ Decouple: Separate the inventory generation (SBOM) from the vulnerability scanning to optimize performance.
    ◦ Context-Aware: Utilize Vulnerability Exploitability eXchange (VEX) logic to downgrade the severity of vulnerabilities that are not practically exploitable in their specific runtime environment.
2. Architectural Blueprint (Local MVP)
• Infrastructure: Native Host OS installations for storage and queuing to simplify the MVP.
    ◦ RustFS: Stores the Software Bill of Materials (SBOM) JSON files, acting as a high-performance artifact cache.
    ◦ PostgreSQL: Stores actionable risk data and vulnerability metadata to enable fast querying.
    ◦ RabbitMQ: Acts as the asynchronous message broker to handle high-throughput scan requests.
• Compute (Workers): Dockerized Python workers that pull jobs from the queue, generate SBOMs, and execute the analysis.
• Vulnerability Database Sync: Vunnel runs locally to aggregate upstream security bulletins (Alpine, NVD, Ubuntu, etc.) into a local SQLite database, preventing rate-limiting and enabling offline scanning.
3. Data Flow: The "Happy Path" Walkthrough
• Step A (Ingestion): A simulated Python emitter.py pushes an image digest (e.g., sha256:...) and its mock runtime context (e.g., {"is_exposed": false}) to the RabbitMQ queue.
• Step B (Inventory Check): The Python worker pulls the message and checks RustFS. If an SBOM does not exist, it runs Syft to extract the package name, version, and license information, and saves the JSON output to RustFS.
• Step C (Analysis): The worker uses Grype to scan the cached SBOM JSON against the local vulnerability database generated by Vunnel.
• Step D (The "Brain" / VEX Logic): The worker evaluates the raw Common Vulnerability Scoring System (CVSS) data against the context payload. Rule: If CVSS is Critical but is_exposed is False, automatically downgrade the effective severity to High and apply a VEX suppression status.
• Step E (Storage & Visualization): Filtered, actionable results are written to PostgreSQL.
4. Discussion Topics & Design Decisions
• Why Image Digests instead of Tags? Image tags (like :latest) are mutable and change over time. Using the SHA256 digest guarantees we are scanning the exact, immutable artifact.
• Why use VEX? A vulnerability (CVE) represents a flaw, but an exposure describes whether it is actually reachable. VEX frameworks provide a machine-readable way to justify why a specific vulnerability does not impact the system, drastically reducing false positives.
• Why skip Kubernetes for the MVP? To rapidly validate the core logic (Queue → SBOM → Scan → VEX → DB) without the overhead of Role-Based Access Control (RBAC) and complex cluster networking.
